9.3 Estilo Purrr

Antes de continuar explorando m√°s variantes de mapas, echemos un vistazo r√°pido a c√≥mo tiende a usar varias funciones purrr para resolver un problema moderadamente realista: ajustar un modelo a cada subgrupo y extraer un coeficiente del modelo. Para este ejemplo de juguete, voy a dividir el conjunto de datos mtcars en grupos definidos por el n√∫mero de cilindros, utilizando la funci√≥n base split:

Esto crea una lista de tres data frames: los autom√≥viles con 4, 6 y 8 cilindros respectivamente.

Ahora imagine que queremos ajustar un modelo lineal, luego extraiga el segundo coeficiente (es decir, la pendiente). El siguiente c√≥digo muestra c√≥mo puede hacer eso con purrr:

by_cyl |> 
  map(~ lm(mpg ~ wt, data = .x)) |> 
  map(coef) |> 
  map_dbl(2)
#>     4     6     8 
#> -5.65 -2.78 -2.19

(Si no ha visto |>, la canalizaci√≥n, antes, se describe en la Secci√≥n 6.3.)

Creo que este c√≥digo es f√°cil de leer porque cada l√≠nea encapsula un solo paso, puedes distinguir f√°cilmente lo funcional de lo que hace, y los ayudantes purrr nos permiten describir de manera muy concisa qu√© hacer en cada paso.

¬øC√≥mo atacar√≠as este problema con la base R? Ciertamente podr√≠a reemplazar cada funci√≥n purrr con la funci√≥n base equivalente:

by_cyl |>  
  lapply(function(data) lm(mpg ~ wt, data = data)) |> 
  lapply(coef) |> 
  vapply(function(x) x[[2]], double(1))
#>     4     6     8 
#> -5.65 -2.78 -2.19

O, por supuesto, podr√≠a usar un bucle for:

slopes <- double(length(by_cyl))
for (i in seq_along(by_cyl)) {
  model <- lm(mpg ~ wt, data = by_cyl[[i]])
  slopes[[i]] <- coef(model)[[2]]
}
slopes
#> [1] -5.65 -2.78 -2.19

Es interesante notar que a medida que pasa de purrr a aplicar funciones b√°sicas a bucles for, tiende a hacer m√°s y m√°s en cada iteraci√≥n. En purrr iteramos 3 veces (map(), map(), map_dbl()), con funciones apply iteramos dos veces (lapply(), vapply()), y con un for loop iteramos una vez. Prefiero m√°s pasos, pero m√°s simples, porque creo que hace que el c√≥digo sea m√°s f√°cil de entender y luego modificar.

9.4 Variantes de map

Hay 23 variantes principales de map(). Hasta ahora, ha aprendido acerca de cinco (map(), map_lgl(), map_int(), map_dbl() y map_chr()). Eso significa que tienes 18 (!!) m√°s para aprender. Eso parece mucho, pero afortunadamente el dise√±o de purrr significa que solo necesitas aprender cinco nuevas ideas:

  • Salida del mismo tipo que la entrada con modify()
  • Iterar sobre dos entradas con map2().
  • Iterar con un √≠ndice usando imap()
  • No devuelve nada con walk().
  • Iterar sobre cualquier n√∫mero de entradas con pmap().

La familia de funciones del mapa tiene entradas y salidas ortogonales, lo que significa que podemos organizar toda la familia en una matriz, con entradas en las filas y salidas en las columnas. Una vez que haya dominado la idea en una fila, puede combinarla con cualquier columna; una vez que haya dominado la idea en una columna, puede combinarla con cualquier fila. Esa relaci√≥n se resume en el siguiente cuadro:

List At√≥mico El mismo tipo Nada
Un argumento map() map_lgl(), ‚Ä¶ modify() walk()
Dos argumentos map2() map2_lgl(), ‚Ä¶ modify2() walk2()
Un argumento + √≠ndice imap() imap_lgl(), ‚Ä¶ imodify() iwalk()
N argumentos pmap() pmap_lgl(), ‚Ä¶ ‚Äî pwalk()

9.4.1 Mismo tipo de salida que de entrada: modify()

Imagina que quisieras duplicar cada columna en un data frame. Primero puede intentar usar map(), pero map() siempre devuelve una lista:

df <- data.frame(
  x = 1:3,
  y = 6:4
)

map(df, ~ .x * 2)
#> $x
#> [1] 2 4 6
#> 
#> $y
#> [1] 12 10  8

Si desea mantener la salida como un data frame, puede usar modify(), que siempre devuelve el mismo tipo de salida que la entrada:

modify(df, ~ .x * 2)
#>   x  y
#> 1 2 12
#> 2 4 10
#> 3 6  8

A pesar del nombre, modify() no modifica en su lugar, devuelve una copia modificada, por lo que si desea modificar permanentemente df, debe asignarlo:

df <- modify(df, ~ .x * 2)

Como de costumbre, la implementaci√≥n b√°sica de modify() es simple y, de hecho, es incluso m√°s simple que map() porque no necesitamos crear un nuevo vector de salida; podemos simplemente reemplazar progresivamente la entrada. (El c√≥digo real es un poco complejo para manejar casos extremos con m√°s gracia).

simple_modify <- function(x, f, ...) {
  for (i in seq_along(x)) {
    x[[i]] <- f(x[[i]], ...)
  }
  x
}

En la Secci√≥n 9.6.2 aprender√° sobre una variante muy √∫til de modify(), llamada modify_if(). Esto le permite (p. ej.) solo duplicar columnas num√©ricas de un data frame con modify_if(df, is.numeric, ~ .x * 2).

9.4.2 Dos entradas: map2() y amigos

map() se vectoriza sobre un √∫nico argumento, .x. Esto significa que solo var√≠a .x cuando se llama a .f, y todos los dem√°s argumentos se pasan sin cambios, por lo que no es adecuado para algunos problemas. Por ejemplo, ¬øc√≥mo encontrar√≠a una media ponderada cuando tiene una lista de observaciones y una lista de pesos? Imagina que tenemos los siguientes datos:

xs <- map(1:8, ~ runif(10))
xs[[1]][[1]] <- NA
ws <- map(1:8, ~ rpois(10, 5) + 1)

Puedes usar map_dbl() para calcular las medias no ponderadas:

map_dbl(xs, mean)
#> [1]    NA 0.463 0.551 0.453 0.564 0.501 0.371 0.443

Pero pasar ws como argumento adicional no funciona porque los argumentos despu√©s de .f no se transforman:

map_dbl(xs, weighted.mean, w = ws)
#> Error in `map_dbl()`:
#> ‚Ñπ In index: 1.
#> Caused by error in `weighted.mean.default()`:
#> ! 'x' and 'w' must have the same length

Necesitamos una nueva herramienta: un map2(), que se vectoriza sobre dos argumentos. Esto significa que tanto .x como .y var√≠an en cada llamada a .f:

map2_dbl(xs, ws, weighted.mean)
#> [1]    NA 0.451 0.603 0.452 0.563 0.510 0.342 0.464

Los argumentos de map2() son ligeramente diferentes a los argumentos de map() ya que dos vectores vienen antes de la funci√≥n, en lugar de uno. Los argumentos adicionales todav√≠a van despu√©s:

map2_dbl(xs, ws, weighted.mean, na.rm = TRUE)
#> [1] 0.504 0.451 0.603 0.452 0.563 0.510 0.342 0.464

La implementaci√≥n b√°sica de map2() es simple y bastante similar a la de map(). En lugar de iterar sobre un vector, iteramos sobre dos en paralelo:

simple_map2 <- function(x, y, f, ...) {
  out <- vector("list", length(x))
  for (i in seq_along(x)) {
    out[[i]] <- f(x[[i]], y[[i]], ...)
  }
  out
}

Una de las grandes diferencias entre map2() y la funci√≥n simple anterior es que map2() recicla sus entradas para asegurarse de que tengan la misma longitud:

En otras palabras, map2(x, y, f) autom√°ticamente se comportar√° como map(x, f, y) cuando sea necesario. Esto es √∫til al escribir funciones; en las secuencias de comandos, generalmente solo usar√≠a la forma m√°s simple directamente.

La base equivalente m√°s cercana a map2() es Map(), que se analiza en la Secci√≥n 9.4.5.

9.4.3 Sin salidas: walk() y amigos

La mayor√≠a de las funciones se llaman por el valor que devuelven, por lo que tiene sentido capturar y almacenar el valor con una funci√≥n map(). Pero algunas funciones se llaman principalmente por sus efectos secundarios (por ejemplo, cat(), write.csv() o ggsave()) y no tiene sentido capturar sus resultados. Toma este ejemplo simple que muestra un mensaje de bienvenida usando cat(). cat() devuelve NULL, as√≠ que mientras map() funciona (en el sentido de que genera las bienvenidas deseadas), tambi√©n devuelve list(NULL, NULL).

welcome <- function(x) {
  cat("Welcome ", x, "!\n", sep = "")
}
names <- c("Hadley", "Jenny")

# Adem√°s de generar las bienvenidas, tambi√©n muestra
# el valor de retorno de cat()
map(names, welcome)
#> Welcome Hadley!
#> Welcome Jenny!
#> [[1]]
#> NULL
#> 
#> [[2]]
#> NULL

Podr√≠as evitar este problema asignando los resultados de map() a una variable que nunca usas, pero que enturbiar√≠a la intenci√≥n del c√≥digo. En su lugar, purrr proporciona la familia de funciones walk que ignoran los valores de retorno de .f y en su lugar devuelven .x de forma invisible1.

#> Welcome Hadley!
#> Welcome Jenny!

Mi representaci√≥n visual de caminar intenta capturar la importante diferencia con map(): las salidas son ef√≠meras y la entrada se devuelve de forma invisible.

Una de las variantes de walk() m√°s √∫tiles es walk2() porque un efecto secundario muy com√∫n es guardar algo en el disco, y cuando guardas algo en el disco siempre tienes un par de valores: el objeto y la ruta que en el que desea guardarlo.

Por ejemplo, imagina que tienes una lista de data frames (que he creado aqu√≠ usando split()) y te gustar√≠a guardar cada uno en un archivo CSV separado. Eso es f√°cil con walk2():

dir(temp)
#> [1] "cyl-4.csv" "cyl-6.csv" "cyl-8.csv"

Aqu√≠ el walk2 () es equivalente a write.csv(cyls[[1]], paths[[1]]), write.csv(cyls[[2]], paths[[2]]), write.csv(cyls[[3]], paths[[3]]).

No existe una base equivalente a walk(); envuelva el resultado de lapply() en invisible() o gu√°rdelo en una variable que nunca se use.

9.4.4 Iterando sobre valores e √≠ndices

Hay tres formas b√°sicas de recorrer un vector con un bucle for:

  • Bucle sobre los elementos: for (x in xs)
  • Bucle sobre los √≠ndices num√©ricos: for (i in seq_along(xs))
  • Bucle sobre los nombres: for (nm in names(xs))

La primera forma es an√°loga a la familia map(). Las formas segunda y tercera son equivalentes a la familia imap() que le permite iterar sobre los valores y los √≠ndices de un vector en paralelo.

imap() es como map2() en el sentido de que su .f se llama con dos argumentos, pero aqu√≠ ambos se derivan del vector. imap(x, f) es equivalente a map2(x, nombres(x), f) si x tiene nombres, y map2(x, seq_along(x), f) si no los tiene.

imap() suele ser √∫til para construir etiquetas:

imap_chr(iris, ~ paste0("The first value of ", .y, " is ", .x[[1]]))
#>                             Sepal.Length 
#> "The first value of Sepal.Length is 5.1" 
#>                              Sepal.Width 
#>  "The first value of Sepal.Width is 3.5" 
#>                             Petal.Length 
#> "The first value of Petal.Length is 1.4" 
#>                              Petal.Width 
#>  "The first value of Petal.Width is 0.2" 
#>                                  Species 
#>   "The first value of Species is setosa"

Si el vector no tiene nombre, el segundo argumento ser√° el √≠ndice:

x <- map(1:6, ~ sample(1000, 10))
imap_chr(x, ~ paste0("The highest value of ", .y, " is ", max(.x)))
#> [1] "The highest value of 1 is 975" "The highest value of 2 is 915"
#> [3] "The highest value of 3 is 982" "The highest value of 4 is 955"
#> [5] "The highest value of 5 is 971" "The highest value of 6 is 696"

imap() es una ayuda √∫til si desea trabajar con los valores de un vector junto con sus posiciones.

9.4.5 Cualquier n√∫mero de entradas: pmap () y amigos

Ya que tenemos map() y map2(), podr√≠as esperar map3(), map4(), map5(), ‚Ä¶ Pero, ¬ød√≥nde te detendr√≠as? En lugar de generalizar map2() a un n√∫mero arbitrario de argumentos, purrr adopta un rumbo ligeramente diferente con pmap(): le proporciona una sola lista, que contiene cualquier n√∫mero de argumentos. En la mayor√≠a de los casos, ser√° una lista de vectores de igual longitud, es decir, algo muy similar a un data frame. En los diagramas, enfatizar√© esa relaci√≥n dibujando la entrada de forma similar a un data frame.

Hay una equivalencia simple entre map2() y pmap(): map2(x, y, f) es lo mismo que pmap(list(x, y), f). El pmap() equivalente a map2_dbl(xs, ws,weighted.mean) utilizado anteriormente es:

pmap_dbl(list(xs, ws), weighted.mean)
#> [1]    NA 0.451 0.603 0.452 0.563 0.510 0.342 0.464

Como antes, los argumentos variables vienen antes de .f (aunque ahora deben estar envueltos en una lista), y los argumentos constantes vienen despu√©s.

pmap_dbl(list(xs, ws), weighted.mean, na.rm = TRUE)
#> [1] 0.504 0.451 0.603 0.452 0.563 0.510 0.342 0.464

Una gran diferencia entre pmap() y las otras funciones de mapa es que pmap() te da un control mucho m√°s preciso sobre la coincidencia de argumentos porque puedes nombrar los componentes de la lista. Volviendo a nuestro ejemplo de la Secci√≥n 9.2.5, donde quer√≠amos cambiar el argumento trim a x, podr√≠amos usar pmap():

trims <- c(0, 0.1, 0.2, 0.5)
x <- rcauchy(1000)

pmap_dbl(list(trim = trims), mean, x = x)
#> [1] -6.6740  0.0210  0.0235  0.0151

Creo que es una buena pr√°ctica nombrar los componentes de la lista para dejar muy claro c√≥mo se llamar√° a la funci√≥n.

A menudo es conveniente llamar a pmap() con un data frame. Una forma pr√°ctica de crear ese data frame es con tibble::tribble(), que le permite describir un data frame fila por fila (en lugar de columna por columna, como de costumbre): pensando en los par√°metros a una funci√≥n como data frame es un patr√≥n muy poderoso. El siguiente ejemplo muestra c√≥mo puede dibujar n√∫meros uniformes aleatorios con diferentes par√°metros:

params <- tibble::tribble(
  ~ n, ~ min, ~ max,
   1L,     0,     1,
   2L,    10,   100,
   3L,   100,  1000
)

pmap(params, runif)
#> [[1]]
#> [1] 0.332
#> 
#> [[2]]
#> [1] 53.5 47.6
#> 
#> [[3]]
#> [1] 231 715 515

Aqu√≠, los nombres de las columnas son fundamentales: eleg√≠ cuidadosamente hacerlos coincidir con los argumentos de runif(), por lo que pmap(params, runif) es equivalente a runif(n = 1L, min = 0, max = 1), runif(n = 2, min = 10, max = 100), runif(n = 3L, min = 100, max = 1000). (Si tiene un data frame en la mano y los nombres no coinciden, use dplyr::rename() o similar).

Hay dos equivalentes base para la familia pmap(): Map() y mapply(). Ambos tienen importantes inconvenientes:

  • Map() vectoriza sobre todos los argumentos para que no pueda proporcionar argumentos que no var√≠en.

  • mapply() es la versi√≥n multidimensional de sapply(); conceptualmente, toma la salida de Map() y la simplifica si es posible. Esto le da problemas similares a sapply(). No existe un equivalente de m√∫ltiples entradas de vapply().

9.5 Familia reduce

Despu√©s de la familia map, la siguiente familia de funciones m√°s importante es la familia reduce. Esta familia es mucho m√°s peque√±a, con solo dos variantes principales, y se usa con menos frecuencia, pero es una idea poderosa, nos brinda la oportunidad de analizar algo de √°lgebra √∫til y potencia el marco de reducci√≥n de mapas que se usa con frecuencia para procesar conjuntos de datos muy grandes.

9.5.1 Lo esencial

reduce() toma un vector de longitud n y produce un vector de longitud 1 llamando a una funci√≥n con un par de valores a la vez: reduce(1:4, f) es equivalente a f(f(f(1, 2), 3), 4).

reduce() es una forma √∫til de generalizar una funci√≥n que funciona con dos entradas (una funci√≥n binaria) para que funcione con cualquier cantidad de entradas. Imagina que tienes una lista de vectores num√©ricos y quieres encontrar los valores que ocurren en cada elemento. Primero generamos algunos datos de muestra:

l <- map(1:4, ~ sample(1:10, 15, replace = T))
str(l)
#> List of 4
#>  $ : int [1:15] 7 1 8 8 3 8 2 4 7 10 ...
#>  $ : int [1:15] 3 1 10 2 5 2 9 8 5 4 ...
#>  $ : int [1:15] 6 10 9 5 6 7 8 6 10 8 ...
#>  $ : int [1:15] 9 8 6 4 4 5 2 9 9 6 ...

Para resolver este desaf√≠o necesitamos usar intersect() repetidamente:

out <- l[[1]]
out <- intersect(out, l[[2]])
out <- intersect(out, l[[3]])
out <- intersect(out, l[[4]])
out
#> [1] 8 4

reduce() Automatiza esta soluci√≥n para nosotros, para que podamos escribir:

reduce(l, intersect)
#> [1] 8 4

Podr√≠amos aplicar la misma idea si quisi√©ramos listar todos los elementos que aparecen en al menos una entrada. Todo lo que tenemos que hacer es cambiar de intersect() a union():

reduce(l, union)
#>  [1]  7  1  8  3  2  4 10  5  9  6

Al igual que la familia de mapas, tambi√©n puede pasar argumentos adicionales. intersect() y union() no aceptan argumentos adicionales, as√≠ que no puedo demostrarlos aqu√≠, pero el principio es sencillo y le hice un dibujo.

Como de costumbre, la esencia de reduce() se puede reducir a un simple envoltorio alrededor de un bucle for:

simple_reduce <- function(x, f) {
  out <- x[[1]]
  for (i in seq(2, length(x))) {
    out <- f(out, x[[i]])
  }
  out
}

El equivalente b√°sico es Reduce(). Tenga en cuenta que el orden de los argumentos es diferente: la funci√≥n viene primero, seguida del vector, y no hay forma de proporcionar argumentos adicionales.

9.5.2 accumulate

La primera variante reduce(), accumulate(), es √∫til para comprender c√≥mo funciona reduce, porque en lugar de devolver solo el resultado final, tambi√©n devuelve todos los resultados intermedios:

accumulate(l, intersect)
#> [[1]]
#>  [1]  7  1  8  8  3  8  2  4  7 10 10  3  7 10 10
#> 
#> [[2]]
#> [1]  1  8  3  2  4 10
#> 
#> [[3]]
#> [1]  8  4 10
#> 
#> [[4]]
#> [1] 8 4

Otra forma √∫til de entender reduce es pensar en sum(): sum(x) es equivalente a x[[1]] + x[[2]] + x[[3]] + ..., es decir reduce(x, `+`). Entonces accumulate(x, `+`) es la suma acumulada:

x <- c(4, 3, 10)
reduce(x, `+`)
#> [1] 17

accumulate(x, `+`)
#> [1]  4  7 17

9.5.3 Tipos de salida

En el ejemplo anterior usando +, ¬øqu√© deber√≠a devolver reduce() cuando x es corto, es decir, longitud 1 o 0? Sin argumentos adicionales, reduce() solo devuelve la entrada cuando x tiene una longitud de 1:

reduce(1, `+`)
#> [1] 1

Esto significa que reduce() no tiene forma de verificar que la entrada sea v√°lida:

reduce("a", `+`)
#> [1] "a"

¬øQu√© pasa si es de longitud 0? Recibimos un error que sugiere que necesitamos usar el argumento .init:

reduce(integer(), `+`)
#> Error in `reduce()`:
#> ! Must supply `.init` when `.x` is empty.

¬øQu√© deber√≠a ser .init aqu√≠? Para averiguarlo, necesitamos ver qu√© sucede cuando se proporciona .init:

As√≠ que si llamamos a reduce(1, `+`, init) el resultado ser√° `1 + init. Ahora sabemos que el resultado deber√≠a ser solo 1, lo que sugiere que .init deber√≠a ser 0:

reduce(integer(), `+`, .init = 0)
#> [1] 0

Esto tambi√©n asegura que reduce() verifique que las entradas de longitud 1 sean v√°lidas para la funci√≥n que est√°s llamando:

reduce("a", `+`, .init = 0)
#> Error in .x + .y: non-numeric argument to binary operator

Si quieres ser algebraico al respecto, 0 se llama la identidad de los n√∫meros reales en la operaci√≥n de suma: si agregas un 0 a cualquier n√∫mero, obtienes el mismo n√∫mero. R aplica el mismo principio para determinar qu√© debe devolver una funci√≥n de resumen con una entrada de longitud cero:

sum(integer())  # x + 0 = x
#> [1] 0
prod(integer()) # x * 1 = x
#> [1] 1
min(integer())  # min(x, Inf) = x
#> [1] Inf
max(integer())  # max(x, -Inf) = x
#> [1] -Inf

Si est√° utilizando reduce() en una funci√≥n, siempre debe proporcionar .init. Piense detenidamente qu√© debe devolver su funci√≥n cuando pasa un vector de longitud 0 o 1, y aseg√∫rese de probar su implementaci√≥n.

9.5.4 M√∫ltiples entradas

Muy ocasionalmente necesita pasar dos argumentos a la funci√≥n que est√° reduciendo. 9.5.5 Mapa reducido

Es posible que haya o√≠do hablar de map-reduce, la idea que impulsa la tecnolog√≠a como Hadoop. Ahora puedes ver cu√°n simple y poderosa es la idea subyacente: map-reduce es un mapa combinado con una reducci√≥n. Como un ejemplo simple, imagine calcular la media de un vector muy grande, tan grande que tiene que dividirse entre varias computadoras. 9.6 Funcionales de predicado

Un predicado es una funci√≥n que devuelve un solo TRUE o FALSE, como is.character(), is.null() o all(), y decimos un predicado coincide con un vector si devuelve TRUE.

9.6.1 Lo esencial

Un predicado funcional aplica un predicado a cada elemento de un vector. purrr proporciona siete funciones √∫tiles que se dividen en tres grupos:

  • some(.x, .p) devuelve TRUE si alg√∫n elemento coincide;
    every(.x, .p) devuelve TRUE si todos los elementos coinciden;
    none(.x, .p) devuelve TRUE si ning√∫n elemento coincide.

    Estos son similares a any(map_lgl(.x, .p)), all(map_lgl(.x, .p)) y all(map_lgl(.x, negate(.p))) pero terminar antes de tiempo: some() devuelve TRUE cuando ve el primer TRUE, y cada() y ninguno() devuelve FALSE cuando ven el primer FALSE o TRUE respectivamente .

  • detect(.x, .p) devuelve el valor de la primera coincidencia; detect_index(.x, .p) devuelve la ubicaci√≥n de la primera coincidencia.

  • keep(.x, .p) mantiene todos los elementos coincidentes; discard(.x, .p) suelta todos los elementos coincidentes.

El siguiente ejemplo muestra c√≥mo puede usar estas funciones con un data frame:

#> 'data.frame':    3 obs. of  2 variables:
#>  $ x: int  1 2 3
#>  $ y: chr  "a" "b" "c"
